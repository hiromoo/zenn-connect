## API リファレンスは辞書のように使う

ここまで、Java API リファレンスを読むために必要なオブジェクト指向プログラミングの考え方を学んできました。

最後に、実際に API リファレンスを読むときの流れを整理します。

API リファレンスは、最初から最後まで順番に読むものではありません。
辞書のように、必要な情報を探すために使うものです。

たとえば、`ArrayList` を使っていて次のような疑問が出たとします。

- 要素を追加するにはどうすればよいか
- 要素を削除するにはどうすればよいか
- 指定した値が含まれているか確認できるか
- コンストラクタにはどのような種類があるか
- どの例外が発生する可能性があるか

このような疑問を解決するために、API リファレンスを確認します。

## ページ上部を見る

クラスの API リファレンスを開いたら、まずページ上部を確認します。
`ArrayList` の場合、次のような情報があります。

```java
java.lang.Object
  java.util.AbstractCollection<E>
    java.util.AbstractList<E>
      java.util.ArrayList<E>
```

これは、`ArrayList` の継承関係を表しています。
`ArrayList` は `AbstractList` を継承し、`AbstractList` は `AbstractCollection` を継承し、最終的に `Object` につながっています。

次に、クラス宣言を確認します。

```java
public class ArrayList<E>
extends AbstractList<E>
implements List<E>, RandomAccess, Cloneable, Serializable
```

この宣言から、次のことが分かります。

- `ArrayList` はクラスである
- ジェネリクス（総称型）の型パラメータ `E` を持つ
- `AbstractList<E>` を継承している
- `List<E>` などのインターフェースを実装している

この情報だけでも、`ArrayList` が「リストとして使える具体的なクラス」であることが読み取れます。

## 説明文を読む

次に、クラスの説明文を読みます。
説明文には、そのクラスが何をするためのものか、どのような特徴があるか、使用上の注意点などが書かれています。

`ArrayList` の説明では、可変サイズの配列として `List` インターフェースを実装していることや、要素の追加、取得、削除に関する特徴が説明されています。

説明文は少し長いことがあります。
最初はすべてを細かく読もうとせず、次の点を探すとよいでしょう。

- このクラスは何を表すのか
- どのような用途で使うのか
- 使うときの注意点はあるか
- スレッドセーフかどうか
- null を許容するかどうか

特にコレクション系のクラスでは、順序、重複、null、性能、スレッドセーフ性に関する説明が重要です。

## コンストラクタの一覧を見る

オブジェクトを作りたいときは、コンストラクタの一覧を見ます。

`ArrayList` には、たとえば次のようなコンストラクタがあります。

```java
ArrayList()
ArrayList(int initialCapacity)
ArrayList(Collection<? extends E> c)
```

それぞれの意味は次のように考えられます。

| コンストラクタ | 意味 |
| --- | --- |
| `ArrayList()` | 空のリストを作る |
| `ArrayList(int initialCapacity)` | 初期容量を指定して空のリストを作る |
| `ArrayList(Collection<? extends E> c)` | 既存のコレクションをもとにリストを作る |

通常は、引数なしのコンストラクタから使い始めれば十分です。

```java
List<String> names = new ArrayList<>();
```

既存のリストをコピーしたい場合は、コレクションを受け取るコンストラクタが使えます。

```java
List<String> original = List.of("Alice", "Bob");
List<String> copied = new ArrayList<>(original);
```

## メソッドの一覧を見る

API リファレンスで最もよく見るのは、メソッドの一覧です。
メソッド一覧では、メソッド名、引数、戻り値を確認します。

たとえば、`add` メソッドには次のような形があります。

```java
boolean add(E e)
void add(int index, E element)
```

同じ `add` という名前でも、引数が違います。
第3章で学んだように、このようなメソッドをオーバーロードされたメソッドと呼びます。

```java
List<String> names = new ArrayList<>();

names.add("Alice");      // 末尾に追加
names.add(0, "Bob");     // 0番目に追加
```

API リファレンスを見るときは、メソッド名だけでなく、引数の数と型も確認しましょう。

## 戻り値を確認する

メソッドを使うときは、戻り値の型も重要です。

```java
boolean add(E e)
E get(int index)
int size()
boolean contains(Object o)
```

それぞれ、戻り値の意味は次の通りです。

| メソッド | 戻り値 |
| --- | --- |
| `add(E e)` | 追加によってリストが変化したかどうか |
| `get(int index)` | 指定した位置の要素 |
| `size()` | 要素数 |
| `contains(Object o)` | 指定した要素が含まれているかどうか |

戻り値があるメソッドは、結果を変数に入れたり、条件式で使ったりできます。

```java
List<String> names = new ArrayList<>();
names.add("Alice");

if (names.contains("Alice")) {
    System.out.println("Alice が含まれています");
}
```

一方、戻り値が `void` のメソッドは、値を返しません。
たとえば、`add(int index, E element)` は指定した位置に要素を追加しますが、戻り値はありません。

## 例外を確認する

API リファレンスには、そのメソッドが投げる可能性のある例外も書かれています。

たとえば、`get(int index)` は、存在しない位置を指定すると `IndexOutOfBoundsException` が発生します。

```java
List<String> names = new ArrayList<>();
names.add("Alice");

System.out.println(names.get(1)); // 例外が発生する
```

`names` には 0 番目の要素しかないため、1 番目を取得しようとすると範囲外です。

API リファレンスの「例外」や `Throws` の欄を見ると、どのような条件で例外が発生するかを確認できます。
メソッドを正しく使うためには、この情報がとても重要です。

## 非推奨の表示に注意する

API リファレンスには、非推奨の API が表示されることがあります。
非推奨の API は、将来的に削除される可能性があったり、よりよい代替手段が用意されていたりする API です。

非推奨の API には、`Deprecated` や `@Deprecated` と表示されます。
新しくコードを書くときは、基本的に非推奨の API は避け、説明文に書かれている代替手段を使います。

## まず見る順番

API リファレンスに慣れていないうちは、次の順番で見ると読みやすいです。

1. クラス名とパッケージ名を見る
2. クラス宣言の `extends` と `implements` を見る
3. 説明文で用途を確認する
4. オブジェクトを作るならコンストラクタを見る
5. 使いたい操作に近いメソッドを探す
6. メソッドの引数と戻り値を確認する
7. 例外や注意事項を確認する

たとえば、「`ArrayList` から要素を削除したい」と思ったら、メソッド一覧で `remove` を探します。
すると、複数の `remove` が見つかります。

```java
E remove(int index)
boolean remove(Object o)
```

`remove(int index)` は、位置を指定して削除します。
`remove(Object o)` は、値を指定して削除します。

```java
List<String> names = new ArrayList<>();
names.add("Alice");
names.add("Bob");

names.remove(0);       // 位置で削除
names.remove("Bob");   // 値で削除
```

このように、API リファレンスでは「似た名前のメソッドの違い」を確認することがよくあります。

## 読めない言葉が出てきたとき

API リファレンスには、最初は難しく感じる言葉が出てきます。
たとえば、次のような言葉です。

- generic
- immutable
- synchronized
- thread-safe
- optional operation
- null permitted

すべてを一度に理解する必要はありません。
まずは、コードを書くうえで直接必要な情報から読めば十分です。

ただし、API リファレンスに書かれている注意書きは、バグを避けるために重要なことが多いです。
分からない言葉があっても読み飛ばすだけで終わらせず、必要になったタイミングで調べる習慣を付けるとよいでしょう。

## この本のまとめ

この本では、Java API リファレンスを読むために必要なオブジェクト指向プログラミングの基本を学びました。

- Java API リファレンスは、クラスやメソッドの使い方を調べる資料
- クラスはオブジェクトの設計図
- オブジェクトは状態と振る舞いを持つ
- コンストラクタはオブジェクトの作成と初期化に使う
- メソッドはオブジェクトに対する操作
- 継承により、クラスはスーパークラスの性質を引き継ぐ
- ポリモーフィズムにより、同じ型でも実際のオブジェクトに応じて動作が変わる
- インターフェースは、クラスが持つべき操作や性質の約束を表す
- ジェネリクス（総称型）により、`ArrayList<String>` のように要素の型を指定できる

Java API リファレンスは、最初は情報量が多く見えます。
しかし、クラス、オブジェクト、継承、インターフェース、メソッド、コンストラクタの意味が分かると、どこに何が書かれているのかが少しずつ見えるようになります。

分からない API に出会ったときは、まずページ上部のクラス宣言を見て、次に説明文、コンストラクタ、メソッド一覧を確認してみてください。
API リファレンスを読む力は、Java のコードを読む力と書く力の両方につながります。
