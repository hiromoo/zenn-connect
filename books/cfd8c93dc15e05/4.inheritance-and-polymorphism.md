## API リファレンスに出てくる `extends`

`ArrayList` の API リファレンスを見ると、クラス宣言の部分に次のような情報があります。

```java
public class ArrayList<E>
extends AbstractList<E>
implements List<E>, RandomAccess, Cloneable, Serializable
```

この中の `extends AbstractList<E>` は、`ArrayList` が `AbstractList` を継承していることを表しています。

継承とは、あるクラスの性質を引き継いで、新しいクラスを作る仕組みです。
継承元のクラスをスーパークラス、継承して作られたクラスをサブクラスと呼びます。

この例では、`AbstractList` がスーパークラス、`ArrayList` がサブクラスです。

## 継承の基本

簡単な例で見てみましょう。

```java
class Animal {
    void eat() {
        System.out.println("食べます");
    }
}

class Dog extends Animal {
    void bark() {
        System.out.println("ワン");
    }
}
```

`Dog` クラスは `Animal` クラスを継承しています。
そのため、`Dog` オブジェクトは `Dog` で定義した `bark` メソッドだけでなく、`Animal` で定義した `eat` メソッドも使えます。

```java
Dog dog = new Dog();
dog.eat();  // 食べます
dog.bark(); // ワン
```

継承により、共通する処理をスーパークラスにまとめ、サブクラスで固有の処理を追加できます。

## オーバーライド

サブクラスでは、スーパークラスのメソッドを上書きできます。
これをオーバーライドと呼びます。

```java
class Animal {
    void makeSound() {
        System.out.println("何かの音");
    }
}

class Dog extends Animal {
    @Override
    void makeSound() {
        System.out.println("ワン");
    }
}
```

`Dog` クラスでは、`Animal` クラスの `makeSound` メソッドをオーバーライドしています。

```java
Dog dog = new Dog();
dog.makeSound(); // ワン
```

`@Override` はアノテーションです。
このメソッドがオーバーライドであることをコンパイラに伝えます。
もしメソッド名や引数を間違えていた場合、コンパイルエラーで気づけるため、基本的には付けるようにします。

## ポリモーフィズムとは？

ポリモーフィズムは、日本語では多態性と訳されます。
同じ型として扱っていても、実際のオブジェクトの種類によって動作が変わる性質のことです。

```java
class Animal {
    void makeSound() {
        System.out.println("何かの音");
    }
}

class Dog extends Animal {
    @Override
    void makeSound() {
        System.out.println("ワン");
    }
}

class Cat extends Animal {
    @Override
    void makeSound() {
        System.out.println("ニャー");
    }
}
```

`Dog` も `Cat` も `Animal` を継承しているため、`Animal` 型の変数で扱えます。

```java
Animal animal1 = new Dog();
Animal animal2 = new Cat();

animal1.makeSound(); // ワン
animal2.makeSound(); // ニャー
```

変数の型はどちらも `Animal` ですが、実際に入っているオブジェクトが `Dog` か `Cat` かによって、呼び出される `makeSound` の処理が変わります。

これがポリモーフィズムです。

## クリーンアーキテクチャとの関係

ポリモーフィズムは、クリーンアーキテクチャのような設計の考え方でも重要です。
クリーンアーキテクチャでは、具体的なクラスに直接依存しすぎず、インターフェースなどの抽象に依存することを重視します。

たとえば、データを保存する処理を考えてみます。

```java
interface UserRepository {
    void save(String name);
}

class DatabaseUserRepository implements UserRepository {
    @Override
    public void save(String name) {
        System.out.println("データベースに保存します: " + name);
    }
}

class InMemoryUserRepository implements UserRepository {
    @Override
    public void save(String name) {
        System.out.println("メモリに保存します: " + name);
    }
}
```

使う側のコードは、具体的な `DatabaseUserRepository` ではなく、`UserRepository` というインターフェースに依存できます。

```java
class UserService {
    private final UserRepository repository;

    UserService(UserRepository repository) {
        this.repository = repository;
    }

    void register(String name) {
        repository.save(name);
    }
}
```

この `UserService` は、実際の保存先がデータベースなのか、メモリなのかを知りません。
しかし、`UserRepository` として扱えるオブジェクトであれば、同じように `save` を呼び出せます。

```java
UserService service1 = new UserService(new DatabaseUserRepository());
UserService service2 = new UserService(new InMemoryUserRepository());

service1.register("Alice");
service2.register("Bob");
```

これもポリモーフィズムの活用例です。
実際のオブジェクトを差し替えても、使う側のコードは同じインターフェースを通して操作できます。
クリーンアーキテクチャでは、この性質を利用して、業務ルールの中心部分をデータベースや画面などの具体的な技術から切り離しやすくします。

この考え方は、テストを書くときにも役立ちます。
`UserService` はコンストラクタで `UserRepository` を受け取っているため、テストでは本物のデータベースを使う代わりに、メモリ上で動く実装やテスト用の実装を渡せます。

```java
UserService service = new UserService(new InMemoryUserRepository());
service.register("Alice");
```

このように、必要な依存オブジェクトを外から渡すことを依存性注入と呼びます。
英語では Dependency Injection と呼ばれ、DI と略されることもあります。
依存性注入を使うと、データベースや外部サービスに直接つながなくても、中心となる処理をテストしやすくなります。

## `ArrayList` と継承

`ArrayList` は `AbstractList` を継承しています。
そのため、`AbstractList` で定義された一部の仕組みを引き継ぎながら、配列を使ったリストとして必要な処理を実装しています。

API リファレンスには、「継承されたメソッド」という欄が表示されます。
これは、そのクラス自身で定義しているメソッドだけでなく、スーパークラスから引き継いだメソッドも使えることを表しています。

たとえば、`ArrayList` のページでは、`AbstractList`、`AbstractCollection`、`Object` などから継承したメソッドを確認できます。

最初はすべてを見る必要はありません。
ただし、「このメソッドは `ArrayList` 自身に書かれていないのに、なぜ使えるのか」と思ったときは、継承されたメソッドの欄を見ると理解しやすくなります。

## `Object` クラス

Java のすべてのクラスは、最終的に `Object` クラスを継承しています。
明示的に `extends` を書かないクラスも、暗黙的に `Object` を継承します。

```java
class User {
}
```

この `User` クラスも、実際には `Object` の性質を引き継いでいます。
そのため、次のようなメソッドを持っています。

- `toString()`
- `equals(Object obj)`
- `hashCode()`

たとえば、`System.out.println(list)` と書いたときにリストの中身が表示されるのは、`toString` メソッドが呼び出されているためです。

```java
ArrayList<String> names = new ArrayList<>();
names.add("Alice");

System.out.println(names); // [Alice]
```

`ArrayList` では、リストの中身が分かりやすく表示されるように `toString` の動作が定義されています。

## より一般的な型として扱う

継承やインターフェースを使うと、具体的なクラスのオブジェクトを、より一般的な型で扱えます。

```java
ArrayList<String> arrayList = new ArrayList<>();
List<String> list = arrayList;
```

この例では、`ArrayList` オブジェクトを `List` 型の変数に代入しています。
`List` はクラスではなくインターフェースです。
ここでは、「具体的な実装を、より一般的な型で扱う」例として見てください。

Java API を使うときは、具体的なクラス名ではなく、より一般的な型で変数を宣言することがよくあります。

```java
List<String> names = new ArrayList<>();
names.add("Alice");
names.add("Bob");
```

こうすると、あとから実装を `LinkedList` などに変更しやすくなります。

```java
List<String> names = new LinkedList<>();
```

使う側が `List` に定義されたメソッドだけを使っていれば、具体的な実装クラスを変更しても、コードへの影響を小さくできます。

## この章のまとめ

この章では、継承とポリモーフィズムについて学びました。

- `extends` はクラスの継承を表す
- サブクラスはスーパークラスの性質を引き継ぐ
- オーバーライドにより、サブクラスでメソッドの動作を変更できる
- ポリモーフィズムにより、同じ型でも実際のオブジェクトに応じて動作が変わる
- ポリモーフィズムを使うと、具体的な実装ではなく抽象に依存する設計がしやすくなる
- 依存性注入を使うと、テスト時に実装を差し替えやすくなる
- API リファレンスの「継承されたメソッド」は、スーパークラスから使えるメソッドを表す

次の章では、`ArrayList` のクラス宣言に出てくる `implements List<E>` を理解するために、抽象クラスとインターフェースを学びます。
