## Colony Chat とは

Colony Chat は、興味やコミュニティごとに人が集まり、会話できるグループ型チャットアプリです。

ざっくり言うと、

- グループを作る
- グループに参加する
- グループの中でチャットする
- 必要に応じて小さな子グループを作る

というアプリです。

ただ、単なるチャットアプリというよりは、**コミュニティごとに会話の境界を作る**ことを意識しています。

例えば、同じ趣味のコミュニティでも、

- 全体のお知らせ
- 雑談
- イベント準備
- 運営メンバーだけの相談
- 特定テーマについて話す場所

のように、話題や参加者の範囲を分けたくなることがあります。

Colony Chat では、それをグループの入れ子構造で表現できるようにしています。

---

## できること

主な機能は次のようなものです。

- グループ作成
- グループチャット
- タイムライン
- 画像付きメッセージ
- 返信スレッド
- グループ招待リンク
- プッシュ通知
- プロフィール作成
- 複数プロフィール
- グループの公開範囲設定
- 通報、ブロック、グループ退出などの安全管理

グループの公開範囲は、次の 3 種類にしています。

| 公開範囲 | 内容 |
| --- | --- |
| オープン | 誰でも見つけられて、すぐ参加できる |
| 公開 | 見つけることはできるが、参加には申請が必要 |
| プライベート | メンバーまたは招待された人だけが見られる |

この 3 種類を、親子グループの構造と組み合わせられるようにしています。

例えば、大きなコミュニティ自体は公開にしておき、その中にプライベートな運営グループを作る、といった使い方ができます。

## なぜ作ったか

作りたかったのは、巨大なタイムラインにすべてが流れていく SNS ではなく、**コミュニティ単位で会話の居場所を作れるアプリ**でした。

今の SNS は、広く届くことに強い一方で、話す場所の文脈が混ざりやすいと感じています。

同じ人でも、仕事の話をするとき、趣味の話をするとき、地域コミュニティで話すときでは、少しずつ振る舞いが違います。

でも、1 つのアカウント、1 つのプロフィール、1 つのタイムラインに集約されると、その違いを表現しにくくなります。

そこで Colony Chat では、1 つのアカウントで複数のプロフィールを持てるようにしました。

これは、匿名性を高めたいというよりも、

> 場ごとに適切なプロフィールで参加できるようにしたい

という意図です。

趣味のコミュニティでは趣味用のプロフィール、仕事寄りのコミュニティでは実名に近いプロフィール、というように、コミュニティごとに自分の見せ方を変えられます。

## 全体アーキテクチャ

技術構成は大きく分けると、次のようになっています。

```text
Flutter App / Web
  ├─ Riverpod
  ├─ go_router
  └─ OpenAPI Generated Dart Client
        ↓
Colony API
  ├─ Hono
  ├─ TypeScript
  ├─ Zod
  └─ OpenAPI
        ↓
Supabase
  ├─ Auth
  ├─ Postgres
  └─ Storage

Firebase
  ├─ Cloud Messaging
  ├─ Analytics
  ├─ Crashlytics
  └─ Performance
```

フロントエンドは Flutter です。
iOS / Android / Web を 1 つのコードベースで出しています。

状態管理は Riverpod、ルーティングは go_router を使っています。
認証状態やプロフィール選択、グループ一覧、未読数、タイムラインなどを Riverpod の provider として扱っています。

バックエンドは TypeScript で、API フレームワークには Hono を使っています。
データベースや認証は Supabase です。

また、API のスキーマは Zod で定義し、OpenAPI を生成しています。
Flutter 側では、その OpenAPI から Dart の Dio クライアントを生成して使っています。

この構成にしている理由は、フロントエンドとバックエンドの間で、リクエストやレスポンスの形がズレるのをできるだけ減らしたかったからです。

個人開発だと、フロントもバックエンドも自分で触ることになります。
その分、勢いで API を変えてしまいがちです。

OpenAPI を間に置いておくと、

- API の一覧を見られる
- Swagger UI で動作確認できる
- Flutter 側のクライアントを生成できる
- 型のズレに気づきやすい

というメリットがあります。

## 設計で意識したこと

### プロフィール単位で参加する

Colony Chat では、グループへの参加やメッセージ投稿は、アカウント単位ではなくプロフィール単位にしています。

内部的には、

- アカウント
- プロフィール
- グループ
- メンバーシップ
- メッセージ

のような関係になっています。

アカウントは認証の単位です。
一方で、コミュニティ上で見える存在はプロフィールです。

この分離によって、1 つのアカウントで複数のコミュニティに違うプロフィールとして参加できるようにしています。

もちろん、この設計にすると実装は少し複雑になります。

例えば、API では「このプロフィールは本当にこのログインユーザーのものか」を毎回確認する必要があります。
また、グループ参加や投稿も、アカウントではなくプロフィールを基準に認可しなければいけません。

ただ、アプリのコンセプト上、ここは最初から分けておきたいポイントでした。

### 階層グループと公開範囲

もう 1 つ大きいのが、グループの階層構造です。

グループは親グループを持つことができます。

```text
コミュニティ
  ├─ 雑談
  ├─ イベント
  │   └─ スタッフ用
  └─ 運営
```

このとき、単に `parent_group_id` を持たせるだけでは不十分です。

例えば、子グループがオープンでも、親グループがプライベートなら、親を見られない人が子だけ見えてしまうのはおかしいです。

そのため、グループを表示できるかどうかは、対象グループだけでなく、親グループをさかのぼって判定するようにしています。

考え方としては、

- private な祖先グループがある場合、そこに参加しているか、招待されている必要がある
- open なグループにはすぐ参加できる
- public なグループには参加申請が必要
- 子グループに参加するには、原則として親グループに参加している必要がある

というルールです。

このあたりは、前回の記事で書いた「認可は画面上でボタンを出さないだけでは不十分」という話ともつながっています。

フロントエンドでボタンを隠すことはできます。
しかし、それだけだと API を直接叩かれたときに守れません。

なので、最終的な認可は API 側で必ず見るようにしています。

### API を信頼境界にする

Flutter 側では、ユーザーにとって自然な UI になるように、

- 参加できるグループには参加ボタンを出す
- 申請が必要なグループには申請ボタンを出す
- 見えないグループは一覧に出さない

といった制御をしています。

ただし、これはあくまで UX のための制御です。

実際にグループへ参加できるか、メッセージを投稿できるか、メンバーを管理できるかは API 側で判定します。

コーディングエージェントを使って開発していると、見た目上動くところまではかなり速く作れます。
一方で、こういう認可境界は人間側で明確に決めておかないと危ないです。

今回の開発でも、ここはかなり意識しました。

### 生成クライアントで型のズレを減らす

Flutter 側から API を呼ぶ部分は、OpenAPI から生成した Dart クライアントを使っています。

API 側で Zod schema を定義し、それを OpenAPI として公開し、Flutter 側で Dio クライアントを生成する流れです。

```text
Zod Schema
  ↓
OpenAPI JSON
  ↓
dart-dio client
  ↓
Flutter app
```

これによって、API の変更を Flutter 側に反映しやすくなります。

もちろん、生成クライアントを使えばすべて解決するわけではありません。
nullable な値の扱いや、JSON Merge Patch のような細かい仕様では、生成コードだけでは表現しづらい部分もありました。

それでも、個人開発でフロントとバックエンドを同時に進める上では、API の契約を明示できるメリットが大きかったです。

## リリースしてみて

今回、Web だけでなく iOS / Android でも公開しました。

実際にストアへ出してみると、アプリとして成立させるためには、機能実装以外にもやることが多いと感じました。

例えば、

- アプリ名やアイコン
- スクリーンショット
- プライバシーポリシー
- 利用規約
- 子どもの安全基準
- 通知の許可導線
- ディープリンク
- アカウント削除
- 通報、ブロック
- クラッシュ検知

などです。

チャットができるだけなら、実装自体はかなり早く進みます。

ただ、実際に人に使ってもらうアプリとして出すなら、安心して使えること、問題が起きたときに対応できること、ストアの要件を満たすことも必要になります。

このあたりは、前回の記事で書いたことを改めて感じた部分でもあります。

コーディングエージェントは実装をかなり助けてくれます。
実際、今回の Colony Chat も AI にかなり手伝ってもらいました。

ただ、

- どういうアプリにするのか
- どの単位で認可するのか
- どこを API 側で保証するのか
- 何をリリース前に確認するのか
- ストアに出すために何が必要なのか

といった判断は、やはり人間側に残ります。

AI と一緒に作ることで、実装速度はかなり上がりました。
一方で、アプリとして成立させるには、設計や検証の重要性も同じくらい上がったように感じています。
